# Intel report template

The full markdown structure for the `intelMarkdown` field written via `skillRun.complete`. The `/prospects/[id]` Intel tab renders this verbatim.

The report has nine sections in this order. Sections that produce no findings are still shown with a one-line "No findings" note plus the queries tried — operators rely on the structure being predictable.

## Voice + format rules

- UK English. No em dashes. No rule-of-three. No flowery openings. (Per `../../CONVENTIONS.md`.)
- Use HTML hyperlinks for URLs that link to evidence: `<a href="https://...">label</a>`. Plain markdown links render unreliably in some HubSpot exports.
- Cite every claim. CH filings by filing ID. Website claims by URL + scrape timestamp. Web findings by URL + retrieved date.
- Numbers in GBP unless explicitly cross-border. Dates in ISO `YYYY-MM-DD`.
- Confidence levels: HIGH (multiple independent sources), MED (single source corroborated by context), LOW (single source, no corroboration).
- Heading levels: `#` for the title, `##` for the nine sections, `###` for sub-headings.

### Report-as-standalone-artefact rule

**Never reference another prospect, client, lender, or skillRun in this report.** The report is shown to lenders, attached to credit memos, sometimes shared with the prospect themselves. A line like "materially stronger than {other company}" or "similar to {other prospect}'s pattern" leaks one entity's data into another's artefact, creates a confidentiality breach in any of those settings, and makes the report context-dependent rather than standalone.

Patterns to avoid:
- "Materially stronger / weaker than {other prospect name}" — even when comparing to another prospect of yours
- "Unlike {other company}" — even as a stylistic device
- "Similar bridge-to-term pattern to {other entity}" — even when the comparison is technically apt
- Referring to a different skillRun's findings or version ("v2 of this report said X")

What to do instead:
- State the finding on its own merits with evidence ("11 outstanding charges across 6 properties" stands on its own)
- Use ARCHETYPE language for patterns ("textbook bridge-to-term refinance pattern" not "matches Mccarthy's pattern")
- If you want to flag versioning context, put it in the front-matter blockquote only, never in the body sections

## When to use HIGH vs MED vs LOW confidence

- **HIGH:** Two or more independent sources agree. Example: Company website states 42 completed units AND planning portal shows 42 approved units across 6 schemes for the company → HIGH.
- **MED:** Single source, but the source has authority + context corroborates. Example: Website project list shows 8 schemes, no other source verifies but website is professional + content is specific → MED.
- **LOW:** Single weak source, contradictions, or inferred without direct evidence. Example: Estimated portfolio scale based on "looks similar to other developers we know" → LOW. Always flag LOW findings in the Risk flags subsection of section 7.

## When the report should be short

If the company is dormant, dissolved, or genuinely sparse (e.g., recent incorporation with no CH filings beyond the incorporation, no website, no web mentions), the report is allowed to be short. Each section says "No findings" with the queries tried. The Recommended Approach section then says "Not pursuable from public data; operator would need to source the lead's contact via referral or direct outreach."

Do not pad. A short report with "Not pursuable" + explicit gaps is more useful than a long report of fluff.
